# v1.1.0 Build Session — Agents & Playbooks

## Context
Salesforce Workspace is a multi-project workspace for Salesforce consulting. Read
`CLAUDE.md` for full project context. v1.0.0 shipped the workflow engine, consulting
assets, and 8 core skills. This release builds the subagent library and all 8 orchestrated
playbooks that chain agents into consulting workflows.

## Release Setup
```
git checkout -b release/v1.1.0 main
```

## Milestone
`v1.1.0` — 12 issues, 5 parallel lanes (A-E). Agents go in `.claude/agents/consulting/`,
playbooks go in `.claude/playbooks/`, remaining skills go in `.claude/skills/`. Zero file
overlap between lanes.

**IMPORTANT**: All playbook YAML definitions reference agents by name. The agent names are
defined in the contract schema at `.claude/agents/consulting/agent-contract.yaml` (from v1.0.0).
Agents and playbooks can be written simultaneously as long as names match the contract.

## Agent Team — Launch 5 Agents in Parallel

### Agent A: Subagent Library (17 agents)
**Scope:** `.claude/agents/consulting/` (all files EXCEPT `orchestrator.md` and `agent-contract.yaml` which exist from v1.0.0)

**Issues (in order):**
1. **#2** — Consulting subagent library — all 17 specialized agents
   - Read `.claude/agents/consulting/agent-contract.yaml` for the contract schema
   - Read `.claude/consulting-assets/` for discovery templates and risk patterns to embed in agent prompts
   - Read `~/.claude/agents-archive/agents-orchestrator.md` for agent definition patterns
   - Create each agent as a `.md` file in `.claude/agents/consulting/`:

   **Discovery & Requirements Agents:**
   - `discovery-intake.md` — Guided interview agent. Uses AskUserQuestion for structured questions. References `.claude/consulting-assets/discovery-questions.md`. Outputs `intake_notes.md`.
   - `gap-analyst.md` — Compares org state against requirements. Uses /sf-org-analyze output + requirements. Outputs `gap_analysis.md` with gap matrix.
   - `requirements-writer.md` — Converts intake + gap analysis into formal requirements + user stories. Outputs `requirements.md`, `user_stories.md`.
   - `stakeholder-mapper.md` — Identifies stakeholders, roles, influence, concerns. Outputs RACI matrix.

   **Design Agents:**
   - `data-modeler.md` — Designs SF data model. Uses /sf-data-model skill logic. Validates limits. Outputs `data_model.md`, ERD.
   - `automation-mapper.md` — Maps processes to tools. Uses /sf-automation-map skill logic + decision guides. Outputs `automation_map.md`.
   - `security-designer.md` — Designs OWD, sharing, profiles, permsets. Uses /sf-security skill. Outputs `security_model.md`.
   - `integration-architect.md` — Designs integration patterns. Uses /sf-integration skill. Outputs `integration_design.md`.

   **Delivery Agents:**
   - `scope-estimator.md` — Complexity estimation. Uses /sf-estimate skill logic. Outputs `estimate.md`.
   - `sow-builder.md` — SOW generation. Outputs `sow_draft.md` + `.docx` via /docx skill.
   - `status-reporter.md` — Reads git log, translates to business language. Outputs `status_update.md`.
   - `demo-scripter.md` — Generates persona-based demo walkthroughs. Outputs `demo_script.md`.
   - `change-request-analyst.md` — Documents scope changes with impact analysis. Outputs `change_request.md`.

   **Investigation Agents:**
   - `issue-diagnostician.md` — Traces symptoms through debug logs, governor limits, automation. Outputs `diagnosis.md`.
   - `impact-analyzer.md` — Maps blast radius of changes across data model, automation, integrations. Outputs `impact_assessment.md`.

   **Knowledge Agents:**
   - `release-notes-analyst.md` — Filters SF release notes by client relevance. Outputs `release_impact.md`.
   - `cert-prep-coach.md` — Generates practice scenarios from Context7 docs. Uses Ollama for question generation.

   Each agent file must include: role/personality, input/output contract, quality criteria, tool permissions, retry configuration.

**Commit pattern:**
```
feat: consulting subagent library with 17 specialized agents (closes #2)
```

### Agent B: Core Engagement Playbooks (Discovery + Design)
**Scope:** `.claude/playbooks/discovery.yaml`, `.claude/playbooks/design.yaml`

**Issues (in order):**
1. **#18** — /sf-playbook discovery — client discovery & requirements workflow
   - Create `.claude/playbooks/discovery.yaml` — Playbook definition with 6 steps:
     1. `discovery-intake` agent → intake_notes.md (checkpoint after)
     2. `org-analyzer` (optional) → org_analysis.md (gate: complete)
     3. `gap-analyst` → gap_analysis.md (gate: all reqs mapped)
     4. `requirements-writer` → requirements.md, user_stories.md (checkpoint after)
     5. `scope-estimator` → estimate.md (checkpoint after)
     6. `deliverable-assembler` → discovery_package/ directory
   - Define inputs: client_name, engagement_type, org_alias (optional)
   - Define deliverables list
   - Define quality gates per step
   - Define 3 human checkpoints (after intake, after requirements, after estimate)

2. **#19** — /sf-playbook design — solution design workflow
   - Create `.claude/playbooks/design.yaml` — Playbook definition with 5 steps:
     1. `data-modeler` → data_model.md, erd.mermaid (gate: all entities mapped)
     2. `automation-mapper` → automation_map.md (gate: all processes mapped, checkpoint after)
     3. `integration-architect` → integration_design.md (gate: all integrations have patterns)
     4. `security-designer` → security_model.md (gate: all objects have sharing model)
     5. `solution-assembler` → solution_design.md + .docx (checkpoint after)
   - Define inputs: requirements.md, org_analysis.md (optional)
   - Can chain after discovery playbook or run standalone

**Commit pattern:**

### Agent C: Build & Deliver Playbooks
**Scope:** `.claude/playbooks/build.yaml`, `.claude/playbooks/deliver.yaml`

**Issues (in order):**
1. **#20** — /sf-playbook build — implementation sprint workflow
   - Create `.claude/playbooks/build.yaml` — Playbook definition with 6 steps:
     1. `project-scaffolder` → SFDX project + org connected (gate: sf org display succeeds)
     2. `test-writer` (per feature) → test classes committed (gate: tests compile)
     3. `implementer` (per feature) → implementation (gate: tests pass + Code Analyzer clean)
     4. Loop steps 2-3 per feature with dev-QA retry (max 3)
     5. `compliance-checker` → compliance_report.md (gate: zero Critical/High, checkpoint after)
     6. `deployer` → deployment to sandbox (gate: deploy succeeds + tests pass in org)
   - TDD workflow enforced: tests committed before implementation
   - Conventional commits with issue references

2. **#21** — /sf-playbook deliver — handoff & go-live workflow
   - Create `.claude/playbooks/deliver.yaml` — Playbook definition with 5 steps:
     1. `compliance-checker` → final quality gate
     2. `deliverable-generator` → all /sf-handoff outputs (gate: all deliverables generated)
     3. `demo-scripter` → training walkthroughs
     4. `deployment-validator` → deployment validation (checkpoint before go-live)
     5. `knowledge-capturer` → engagement learnings to Obsidian
